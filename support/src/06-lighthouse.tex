\addcontentsline{toc}{part}{\protect\textbf{Faza 6 \textendash{} Lighthouse}}
\handout{Faza 6 \textendash{} Lighthouse}{Audit automatizat. Indicatori de performanta. Core Web Vitals.}

Lighthouse este un instrument de audit automatizat dezvoltat de Google, integrat in Chrome DevTools si in serviciul web PageSpeed Insights. Capitolul prezinta cele patru categorii evaluate, modul de interpretare a raportului, indicatorii \emph{Core Web Vitals} folositi de Google in algoritmul de ranking, si remedierile pentru problemele frecvente.

\bigskip

\begin{outcomes}
Dupa capitolul asta vei \textcommabelow{s}ti:
\begin{itemize}[itemsep=1pt]
  \item Ruleaza un audit Lighthouse pe propria pagina prin PageSpeed Insights sau DevTools.
  \item Cite\textcommabelow{s}te raportul \textcommabelow{s}i identifica problemele cu impact maxim pe scor.
  \item Aplica fix-uri uzuale: \code{alt}, \code{lang}, \code{width}/\code{height} pe imagini, \code{defer} pe script.
  \item Optimizeaza imaginile cu format modern (WebP), \code{srcset}, \code{loading="lazy"}.
  \item Optimizeaza fonturile: \code{font-display: swap}, \code{preload}, subseturi, self-hosting.
  \item In\textcommabelow{t}elege \emph{critical rendering path}: ce blocheaza prima afi\textcommabelow{s}are si cum reduci timpul pana la LCP.
  \item Interpreteaza Core Web Vitals (LCP, INP, CLS) \textcommabelow{s}i pragurile lor.
  \item \faRocket\ \emph{Bonus self-study:} minification, third-party scripts, service worker, Lighthouse CI, \code{content-visibility}.
\end{itemize}
\end{outcomes}

\begin{whymatters}
Lighthouse este standardul industrial pentru evaluarea unei pagini web. Scorul 95+ pe toate cele 4 categorii este criteriul implicit la code review in firmele moderne. Mai mult, Core Web Vitals influenteaza direct ranking-ul Google. Un audit de 2 minute identifica 80\% din problemele care ar costa ore de cautare manuala.
\end{whymatters}

\begin{nota}[Pre-rechizite]
Capitolul presupune un site deployat (Faza 4) si configurat cu meta tags (Faza 5). Auditul ruleaza pe URL-ul public.
\end{nota}

% =========================================================
\section{Categoriile de evaluare}

Lighthouse genereaza un scor 0\textendash{}100 pe fiecare dintre cele patru axe:

\begin{itemize}
  \item \textbf{Performance} \textendash{} timpul de incarcare si receptivitatea paginii.
  \item \textbf{Accessibility} \textendash{} conformitatea cu standardele WCAG (in principal contrast, structura semantica, asocieri label\textendash{}input, atribute \code{alt}).
  \item \textbf{Best Practices} \textendash{} folosirea de protocoale si API-uri moderne, absenta erorilor in consola, lipsa codului depasit.
  \item \textbf{SEO} \textendash{} prezenta meta tag-urilor obligatorii, structura semantica, indexabilitatea.
\end{itemize}

Scorurile se interpreteaza astfel: peste 90 = bun; peste 95 = nivel profesional; sub 50 = necesita interventie.

\subsection{Lab data vs field data}

Lighthouse foloseste \emph{lab data}: un scenariu sintetic in conditii standardizate (CPU lent, retea simulata 4G slow). Scorul reflecta capacitatea pe un dispozitiv mediu, nu situatia reala a utilizatorilor. Pentru date din productie, PageSpeed Insights afiseaza si \emph{field data} extrase din Chrome User Experience Report (CrUX), agregate pe ultimele 28 de zile.

% =========================================================
\section{Modalitati de executie}

\subsection{PageSpeed Insights (interfata web)}

Cea mai accesibila metoda. Pasi: accesarea \url{https://pagespeed.web.dev}, lipirea URL-ului public al paginii, executarea analizei. Avantaje: nu necesita instalare; include datele reale de la utilizatori (CrUX) atunci cand acestea exista; raport partajabil prin URL.

\subsection{Chrome DevTools (local)}

Procedura: deschiderea paginii in Chrome, \code{F12} $\to$ tab \emph{Lighthouse}, selectarea categoriilor, executarea \emph{Analyze page load}. Avantaje: functioneaza pe paginile servite de pe \code{localhost}, fara necesitatea de deploy; permite testare rapida intre modificari.

\subsection{CLI (pentru integrare CI)}

Lighthouse este disponibil ca pachet npm. Folosit in pipeline-uri CI/CD pentru a bloca deploy-urile care scad scorul sub un prag stabilit.

\section{Structura raportului}

Fiecare categorie include:

\begin{itemize}
  \item Scorul numeric, evidentiat color (rosu / portocaliu / verde).
  \item Lista problemelor identificate, grupate pe tip. Selectarea unei probleme afiseaza descrierea, impactul, si pasii de remediere.
  \item Lista verificarilor trecute cu succes.
  \item Sectiunea \emph{Opportunities} (pentru Performance) listeaza imbunatatirile cu economia estimata in milisecunde.
\end{itemize}

\section{Probleme frecvente si remediere}

\subsection{Performance}

\begin{itemize}
  \item \textbf{Imagini fara dimensiuni explicite.} Setarea atributelor \code{width} si \code{height} pe \code{<img>} permite browserului sa rezerve spatiu, prevenind \emph{Cumulative Layout Shift}.
  \item \textbf{Resurse care blocheaza randarea.} Adaugarea atributului \code{defer} pe \code{<script>} permite parsarea HTML in paralel.
  \item \textbf{Imagini supradimensionate.} O imagine 2400\,x\,1600 redata la 800 pixeli iroseste banda. Optimizare: \url{https://squoosh.app}.
  \item \textbf{Format imagini neoptimizat.} JPEG/PNG in locul WebP / AVIF.
  \item \textbf{Cache lipsa.} Resursele statice (CSS, JS, imagini) ar trebui servite cu header \code{Cache-Control: max-age=31536000} (1 an). \emph{Caveat GitHub Pages:} serve\textcommabelow{s}te cu \code{max-age=600} (10 minute) \textcommabelow{s}i nu poate fi modificat \textendash{} Lighthouse va afi\textcommabelow{s}a avertismentul ,,Serve static assets with an efficient cache policy''. Pentru control real al cache-ului, foloseste Netlify sau Cloudflare Pages.
\end{itemize}

\subsection{Accessibility}

\begin{itemize}
  \item \textbf{Atribut \code{alt} lipsa pe \code{<img>}.} Toate imaginile relevante semantic necesita \code{alt}; imaginile pur decorative folosesc \code{alt=""}.
  \item \textbf{Buton fara nume accesibil.} Includerea de text in interior sau adaugarea \code{aria-label}.
  \item \textbf{Etichete lipsa pe formulare.} Asocierea \code{<label for>} cu \code{<input id>}.
  \item \textbf{Contrast insuficient.} Conform WCAG AA, raportul de contrast minim este 4.5:1 pentru text normal si 3:1 pentru text mare. Verificare: \url{https://webaim.org/resources/contrastchecker/}.
\end{itemize}

\subsection{Best Practices}

\begin{itemize}
  \item \textbf{Erori in consola.} Lighthouse penalizeaza orice eroare \code{console.error} la incarcare. Verificare: \code{F12 $\to$ Console}.
  \item \textbf{Resurse incarcate prin HTTP (nu HTTPS).} Mixed content; rezolvat automat pe GitHub Pages.
\end{itemize}

\subsection{SEO}

\begin{itemize}
  \item \textbf{Document fara \code{<title>}.}
  \item \textbf{Lipsa \code{<meta name="description">}.}
  \item \textbf{Element \code{<html>} fara \code{lang}.}
  \item \textbf{Linkuri fara text descriptiv} (\code{<a href="X">click aici</a>} este penalizat).
\end{itemize}

% =========================================================
\section{Core Web Vitals}

Cei trei indicatori folositi explicit de Google in algoritmul de ranking, incepand cu 2021:

\begin{itemize}
  \item \textbf{LCP} (\emph{Largest Contentful Paint}) \textendash{} timpul pana la afisarea celui mai mare element vizibil. Prag: $<$2.5 secunde.
  \item \textbf{INP} (\emph{Interaction to Next Paint}) \textendash{} latenta intre o actiune a utilizatorului si raspunsul vizual. Prag: $<$200 milisecunde.
  \item \textbf{CLS} (\emph{Cumulative Layout Shift}) \textendash{} masura instabilitatii vizuale in timpul incarcarii. Prag: $<$0.1.
\end{itemize}

\section{Optimizarea imaginilor}

Imaginile reprezinta, in medie, 50\% din dimensiunea unei pagini web. Optimizarea lor are impact direct asupra scorului Performance.

\subsection{Format modern}

WebP reduce dimensiunea cu 25\textendash{}35\% fata de JPEG, AVIF cu 50\%. Conversia se face cu \url{https://squoosh.app} (interfata web, fara instalare).

\subsection{Atribute necesare pe \texttt{<img>}}

\begin{dano}[Asa nu]
\begin{lstlisting}[style=html, numbers=none]
<img src="hero.jpg">
\end{lstlisting}
\textbf{Probleme.} Lipseste \code{alt} (a11y); lipsesc \code{width}/\code{height} (CLS); lipseste \code{loading} (download eager); o singura sursa pentru toate ecranele (banda risipita pe mobil).
\end{dano}

\begin{dado}[Asa da \textendash{} optimizat complet]
\begin{lstlisting}[style=html, numbers=none]
<img src="hero-800.webp"
     srcset="hero-400.webp 400w,
             hero-800.webp 800w,
             hero-1600.webp 1600w"
     sizes="(min-width: 768px) 50vw, 100vw"
     width="800" height="450"
     loading="lazy"
     alt="Echipa BEST Iasi in fata sediului UAIC">
\end{lstlisting}
\textbf{Beneficii.} Browserul alege rezolutia potrivita din \code{srcset}; imaginile sub fold sunt incarcate doar la nevoie; rezerva spatiu prin \code{width}/\code{height}; alt asigura accesibilitatea.
\end{dado}

\subsection{Lazy loading}

Atributul \code{loading="lazy"} amana descarcarea imaginilor aflate sub \emph{fold} (zona vizibila initial). Suportul browser este universal incepand cu 2020.

\textbf{Exceptie:} pentru imaginea \emph{hero} (prima imagine din \emph{viewport}), se foloseste \code{loading="eager"} sau atributul lipseste; \code{loading="lazy"} aici intarzie LCP-ul.

\subsection{Squoosh workflow pas cu pas}

\url{https://squoosh.app} face conversia + redimensionarea intr-o singura ses\u iune, in browser (fara upload pe server). Pa\textcommabelow{s}i:

\begin{enumerate}[itemsep=1pt]
  \item Drag-and-drop imaginea originala (JPEG 4000\,x\,3000, 4MB).
  \item In panoul \emph{Resize} (dreapta): seteaza la\textcommabelow{t}imea finala (ex. 1600px pentru hero, 800px pentru carduri).
  \item Schimba formatul de output din PNG/JPEG in \emph{WebP} sau \emph{AVIF}.
  \item Aju\textcommabelow{s}teaza \emph{Quality} (75 e un sweet spot pentru WebP). Compara vizual stanga vs dreapta.
  \item Download. Repeta pentru 3\textendash{}4 dimensiuni daca folose\textcommabelow{s}ti \code{srcset}.
\end{enumerate}

Rezultat tipic: \textbf{500KB JPEG $\to$ 60KB WebP}. Diferen\textcommabelow{t}a de calitate la 75\% e impercepti\-bila pentru ochiul uman, dar Lighthouse vede economia.

\section{Optimizarea fonturilor}

Fonturile custom (Google Fonts, fonturi cumparate) sunt al doilea cost major dupa imagini. Un \code{<h1>} care a\textcommabelow{s}teapta sa se incarce fontul = LCP intarziat. Trei tehnici care rezolva 90\% din probleme.

\subsection{\texttt{font-display: swap}}

Implicit, textul cu font custom e \textbf{invizibil} pana cand fontul se descarca (\emph{Flash of Invisible Text, FOIT}). \code{font-display: swap} il afi\textcommabelow{s}eaza imediat cu fontul fallback, apoi schimba cand fontul custom e gata (\emph{Flash of Unstyled Text, FOUT}).

\begin{lstlisting}[style=css, caption={Self-hosting cu \texttt{font-display}}]
@font-face {
  font-family: 'Lato';
  src: url('/fonts/lato-regular.woff2') format('woff2');
  font-weight: 400;
  font-style: normal;
  font-display: swap;
}
\end{lstlisting}

Pentru Google Fonts, parametrul vine in URL: \code{?family=Lato\&display=swap}.

\subsection{\texttt{preload} pentru fonturi critice}

Browserul descopera fonturile abia dupa ce parseaza CSS-ul si gase\textcommabelow{s}te o regula care le folose\textcommabelow{s}te \textendash{} prea tarziu pentru LCP. \code{preload} le aduce in paralel cu HTML-ul:

\begin{lstlisting}[style=html, numbers=none]
<link rel="preload" href="/fonts/lato-regular.woff2" as="font" type="font/woff2" crossorigin>
\end{lstlisting}

\begin{atentie}
Preload doar fonturile \textbf{folosite in viewport-ul ini\textcommabelow{t}ial}. Preload-ul pentru fonturi rar folosite (ex. font pentru cod, vizibil doar pe pagina de blog) face contrariul: ocupa banda inainte sa fie necesar.
\end{atentie}

\subsection{Subseturi \textcommabelow{s}i self-hosting}

Fontul implicit Lato de la Google include 800+ glife (chineza, japoneza, simboluri). Daca ai nevoie doar de \textbf{latin} (engleza + romana), trimi\textcommabelow{t}i pe wire 5x mai mult decat nevoie. Solu\textcommabelow{t}ii:

\begin{itemize}[itemsep=1pt]
  \item \textbf{Google Fonts URL cu subset:} \code{?family=Lato\&subset=latin-ext\&display=swap}.
  \item \textbf{Self-hosting} cu Google Webfonts Helper: \url{https://gwfh.mranftl.com}. Descarci fi\textcommabelow{s}ierele \code{.woff2} subsetate \textcommabelow{s}i CSS-ul gata configurat, le servesti din site-ul tau.
\end{itemize}

Benefi\textcommabelow{t}ii self-hosting: o solicitare HTTP mai pu\textcommabelow{t}in (nu mai contactezi \code{fonts.googleapis.com}), niciun cost DNS, control complet asupra cache-ului.

\section{Critical rendering path \textendash{} ce blocheaza prima afi\textcommabelow{s}are}

\emph{Critical rendering path} (CRP) este secven\textcommabelow{t}a de pa\textcommabelow{s}i pe care browserul ii face de la primirea HTML-ului pana la afi\textcommabelow{s}area pixelilor pe ecran. Doua resurse blocheaza vizibil acest path:

\begin{itemize}[itemsep=1pt]
  \item \textbf{CSS} \textendash{} blocheaza \emph{render}. Browserul nu deseneaza nimic pana cand nu a parsat tot CSS-ul referit in \code{<head>}. De aceea CSS-ul critic merge \code{<head>}, nu la sfar\textcommabelow{s}itul \code{<body>}.
  \item \textbf{JavaScript fara \code{defer}/\code{async}} \textendash{} blocheaza si \emph{parsing} si \emph{render}. Browserul opre\textcommabelow{s}te totul, descarca, executa, apoi continua.
\end{itemize}

\begin{lstlisting}[style=html, caption={Tipare}]
<head>
  <!-- BLOCANT: dar CSS-ul e critic, e ok aici -->
  <link rel="stylesheet" href="style.css">

  <!-- NEBLOCANT: descarca paralel, executa dupa parsare -->
  <script src="app.js" defer></script>

  <!-- INDEPENDENT (analytics): paralel, executie cand vrea -->
  <script src="https://analytics.com/script.js" async></script>
</head>
\end{lstlisting}

\subsection{De ce \code{defer} e default-ul pentru JS-ul tau}

\begin{itemize}[itemsep=1pt]
  \item \code{defer} \textendash{} descarca in paralel, executa \emph{dupa} parsare HTML, in \textbf{ordinea declarata}. DOM-ul exista cand ruleaza.
  \item \code{async} \textendash{} descarca in paralel, executa \emph{imediat ce e disponibil} (poate intrerupe parsing). Ordine non-determinista. Pentru analytics da, pentru codul tau nu.
  \item Fara nimic \textendash{} blochea\-za totul. Antipattern din 2015 incoace.
\end{itemize}

\subsection{Reducerea FCP / LCP}

\begin{itemize}[itemsep=1pt]
  \item Minimizeaza CSS-ul critic (tot ce e in viewport-ul initial). 50\,KB CSS = 2x mai lent pe 3G.
  \item Preload pentru fontul + imaginea hero.
  \item Inline-uiaza CSS-ul critic in \code{<style>} pe pagini cu trafic mare (landing-uri); restul \code{<link>} async.
  \item Evita render-blocking 3rd party (chat widgets, Google Tag Manager) in \code{<head>}.
\end{itemize}

\section{Imbunatatiri suplimentare pentru Performance}

\subsection{Preload pentru resurse critice}

Resursele critice (font principal, imaginea hero) pot fi declarate cu \code{<link rel="preload">} pentru ca browserul sa le descarce inainte de a parsa CSS-ul.

\begin{lstlisting}[style=html]
<link rel="preload" href="fonts/lato.woff2" as="font" type="font/woff2" crossorigin>
<link rel="preload" href="hero.webp" as="image">
\end{lstlisting}

\subsection{CSS critic inline}

Pentru paginile cu prima vizita prioritara (landing pages), CSS-ul critic (necesar pentru randarea zonei vizibile) poate fi inclus direct in \code{<head>} prin \code{<style>}, restul fiind incarcat asincron. Tehnica nu este necesara pentru pagini personale; merita atentie pentru site-uri comerciale.

\begin{bonusbox}
\textbf{Acest box e optional.} Daca esti la inceput, treci direct la \emph{Verificare}. Mai jos sunt optimiz\u arile pe care le faci cand vrei 100/100/100/100, nu doar 95+.

\medskip

\textbf{\faCompress\ Minification \textendash{} elimini spa\textcommabelow{t}iile, p\u astrezi semnifica\textcommabelow{t}ia}

Codul scris pentru oameni (indentare, comentarii, spa\textcommabelow{t}ii) ocupa 30\textendash{}50\% in plus fa\textcommabelow{t}a de ce trebuie sa primeasca browserul. \emph{Minification} elimina tot ce e doar pentru lizibilitate. Rezultat: fi\textcommabelow{s}ier identic func\textcommabelow{t}ional, mai mic.

Tool-uri online (zero setup):
\begin{itemize}[itemsep=1pt]
  \item \emph{CSS:} \url{https://www.toptal.com/developers/cssminifier}
  \item \emph{JS:} \url{https://www.toptal.com/developers/javascript-minifier}
  \item \emph{HTML:} \url{https://www.willpeavy.com/tools/minifier/}
\end{itemize}

Pentru proiecte reale, foloseste un build tool (Vite, esbuild, Parcel) care minifica automat la build. Pentru un portofoliu pe GitHub Pages, varianta online + commit la finalul fazei e suficient.

\medskip

\textbf{\faBullhorn\ Third-party scripts \textendash{} cel mai mare cost ascuns}

Analytics, chat widgets, embed-uri YouTube/Twitter \textendash{} fiecare adauga 50\textendash{}500ms si penalty pe Best Practices.

Strategii (in ordinea cre\textcommabelow{s}terii impactului):

\begin{itemize}[itemsep=2pt]
  \item \textbf{\code{async} pe analytics:} \code{<script async src="..."></script>}. Default deja pentru Google Analytics modern.
  \item \textbf{Lazy load la interac\textcommabelow{t}iune:} chat widget se incarca \textbf{doar dupa ce user-ul d\u a click}. Pana atunci, un buton placeholder de 1KB.
  \item \textbf{Facade pentru embed-uri:} YouTube embed = 500KB. Inlocuie\textcommabelow{s}te cu o imagine + buton play, incarca iframe-ul la click. Tool: \url{https://github.com/paulirish/lite-youtube-embed}.
  \item \textbf{Self-host pentru analytics:} Plausible/Umami self-hosted in loc de Google Analytics. Zero penalty Lighthouse, GDPR-friendly.
\end{itemize}

\begin{lstlisting}[style=js, caption={Lazy load pentru chat widget}]
const chatBtn = document.querySelector('.chat-launcher');
chatBtn.addEventListener('click', () => {
  const script = document.createElement('script');
  script.src = 'https://widget.intercom.io/widget/abc123';
  document.head.appendChild(script);
}, { once: true });
\end{lstlisting}

\medskip

\textbf{\faWifi\ \texttt{content-visibility: auto} \textendash{} skip rendering pe ce nu e vizibil}

CSS modern (2023+) care permite browserului sa nu randeze elementele aflate sub fold pana cand utilizatorul scroll-eaza catre ele. Impact major pe pagini lungi (blog posts, documentari).

\begin{lstlisting}[style=css, numbers=none]
.long-section { content-visibility: auto; contain-intrinsic-size: 500px; }
\end{lstlisting}

\code{contain-intrinsic-size} e dimensiunea estimata \textendash{} previne \emph{layout shift} cand sec\textcommabelow{t}iunea devine vizibila si se randeaza real.

\medskip

\textbf{\faServer\ Service Worker \textendash{} cache offline (teaser)}

Service Worker este un script care ruleaza in fundal si poate intercepta cererile HTTP \textendash{} il folose\textcommabelow{s}ti pentru a servi fi\textcommabelow{s}iere din cache, oferind func\textcommabelow{t}ionalitate \textbf{offline}. Combinat cu manifest.json, transforma un site in \emph{Progressive Web App} (PWA): instalabil pe telefon, deschis fara internet.

Setup minimal: 30 de linii de JS. Tutorial complet: \url{https://web.dev/learn/pwa}.

Pentru un portofoliu, valoarea e limitata; pentru un blog citit recurent, sau o aplicatie de productivitate, e diferen\textcommabelow{t}a intre ,,site'' si ,,app''.

\medskip

\textbf{\faRobot\ Lighthouse CI \textendash{} audit automat la fiecare commit}

GitHub Actions poate rula Lighthouse la fiecare push si \textbf{blocheaza merge-ul} daca scorul scade sub un prag (ex. 90). Asa nu mai poti accidental regresa.

\begin{lstlisting}[caption={\texttt{.github/workflows/lighthouse.yml} minimal}, basicstyle=\ttfamily\small, backgroundcolor=\color{codeBg}, frame=leftline, framerule=2.5pt, rulecolor=\color{partcolor}, framexleftmargin=4pt, xleftmargin=14pt]
name: Lighthouse CI
on: [pull_request]
jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: treosh/lighthouse-ci-action@v11
        with:
          urls: 'https://voluntar-best.github.io/'
          uploadArtifacts: true
\end{lstlisting}

Setup complet: \url{https://github.com/treosh/lighthouse-ci-action}.

\section*{Recapitulare}

\begin{recap}[Faza 6 -- Lighthouse]
\begin{itemize}[itemsep=1pt]
  \item Patru categorii: Performance, Accessibility, Best Practices, SEO.
  \item Praguri: 90+ acceptabil, 95+ profesional, sub 50 necesita interventie.
  \item Lab data (Lighthouse) vs field data (CrUX); PageSpeed Insights le combina.
  \item Executie: \url{https://pagespeed.web.dev} (web) sau Chrome DevTools $\to$ \emph{Lighthouse}.
  \item Top fix-uri: \code{alt} pe imagini, \code{lang} pe \code{<html>}, contrast $\geq$4.5:1, \code{width}/\code{height} pe \code{<img>}, \code{defer} pe \code{<script>}.
  \item Core Web Vitals: LCP $<$2.5s, INP $<$200ms, CLS $<$0.1.
  \item Imagini: WebP/AVIF, \code{srcset}, \code{loading="lazy"} (cu exceptia imaginii hero), dimensiuni explicite.
  \item Squoosh: drag $\to$ resize $\to$ format WebP $\to$ quality 75 $\to$ download. 500KB $\to$ 60KB tipic.
  \item Fonturi: \code{font-display: swap} previne FOIT; \code{<link rel="preload">} pentru fontul critic; self-host pentru ,,o conexiune mai pu\textcommabelow{t}in''.
